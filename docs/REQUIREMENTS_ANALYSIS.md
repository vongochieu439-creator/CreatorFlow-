# CreatorFlow 需求分析文档

## 1. 项目背景

创作者在公众号、知乎、B站、小红书等平台同步发布内容时，常常面临重复改稿的问题。同一份内容不能直接复制到所有平台，因为不同平台的内容消费场景、推荐机制、标题风格、正文结构和互动方式都不一样。

CreatorFlow 希望解决的问题是：用户只需要输入一份主稿，系统就能根据目标平台自动生成更适合发布的版本，并完成发布前检查和模拟发布。

## 2. 目标用户

### 个人创作者

这类用户通常一人负责选题、写作、发布和数据复盘，时间有限，希望减少重复排版和改稿成本。

### 新媒体运营

这类用户需要把同一活动、产品、观点或内容分发到多个平台，更关注效率、规范性和发布质量。

### 产品运营 / 内容运营

这类用户经常需要把产品更新、活动说明、案例文章改写成不同平台版本，希望内容既准确又符合平台语境。

## 3. 用户痛点

1. 多平台格式差异大  
公众号适合完整长文，知乎重视观点论证，B站需要视频简介和时间轴，小红书更偏短句、清单和标签。

2. 重复劳动多  
用户要反复修改标题、摘要、正文结构、标签、互动引导，耗时且容易疲劳。

3. 缺少平台风格判断  
用户知道要适配平台，但不一定知道每个平台怎样表达更容易被读者接受。

4. 发布前容易漏项  
标题过长、标签缺失、正文过短、互动引导不足等问题经常在发布后才发现。

5. 内容质量不稳定  
用户希望参考平台热门内容的结构和表达方式，但又不能直接复制别人的内容。

## 4. 核心需求

### 4.0 AI 平台爆款分析

在生成多平台版本之前，系统应先分析目标平台近期优质内容的共性规律，形成“平台内容策略”。

用户不是只需要普通改写，而是希望系统回答：

- 这个平台最近什么标题更容易被点击？
- 爆款内容通常怎么开头？
- 正文是清单式、故事式、观点式，还是教程式？
- 标签通常怎么组合？
- 评论区互动通常怎么引导？
- 哪些表达容易显得像搬运或营销？

系统可以通过两种方式实现：

1. 比赛演示版：内置平台爆款特征模板，模拟 AI 分析结果。
2. 真实产品版：接入小米 API 或其他大模型 API，并结合平台公开内容、用户提供的参考链接或历史发布数据进行分析。

分析结果不直接复制爆款内容，而是抽象成结构和策略：

```json
{
  "platform": "小红书",
  "titlePatterns": ["痛点 + 结果", "人群 + 避坑", "数字 + 清单"],
  "openingStyle": "先说痛点，再给结果承诺",
  "bodyStructure": ["适合谁", "核心清单", "个人经验", "互动提问"],
  "tagStrategy": ["人群标签", "场景标签", "问题标签"],
  "riskTips": ["避免夸大承诺", "避免照搬参考内容"]
}
```

### 4.1 一稿多平台适配

用户输入一份主稿后，系统自动生成不同平台版本：

- 公众号：长文结构、方法论总结、收藏价值。
- 知乎：结论先行、逻辑分析、问题回答。
- B站：视频简介、时间轴、互动引导。
- 小红书：短句清单、情绪表达、标签分发。

### 4.2 平台风格仿写

系统可以接入大模型 API，例如小米相关 AI API 或其他兼容的大模型服务，根据平台优质内容的共性特征进行改写。

这里的“仿写”不是复制具体爆款内容，而是提取其结构特征：

- 标题句式
- 开头方式
- 段落节奏
- 信息密度
- 标签组合
- 互动引导方式

系统应避免直接复用他人原创表达，降低抄袭和同质化风险。

### 4.3 发布前检查

系统在生成后给出检查结果：

- 标题是否超长。
- 正文字数是否符合平台习惯。
- 是否生成标签。
- 是否包含互动引导。
- 是否符合平台内容结构。

### 4.4 可编辑与可复制

AI 生成的内容不能直接替用户做最终决定。用户需要能够：

- 查看不同平台版本。
- 手动修改发布稿。
- 复制当前平台稿。
- 导出全部平台稿。
- 恢复自动生成版本。

### 4.5 模拟发布

在比赛作品中可先实现模拟发布：

- 点击一键发布。
- 通过 MockPublisher 展示各平台发布状态。
- 生成发布记录。
- 记录发布时间和平台名称。

后续真实上线时，再接入各平台开放接口或半自动发布流程。

发布器设计应按平台差异拆分：

- `MockPublisher`：比赛 Demo 使用，保证稳定展示完整闭环。
- `WechatDraftPublisher`：对接公众号草稿箱。
- `BilibiliSubmissionPublisher`：对接视频投稿和稿件信息。
- `PlatformSharePublisher`：对接分享式发布能力。
- `ManualAssistPublisher`：对暂未开放接口的平台提供复制、导出、人工确认。

这种设计避免把所有平台强行抽象成同一种发布接口，也能更真实地反映平台权限和风控差异。

### 4.6 人工确认、直接编辑与版本回退

AI 生成内容不能直接替用户发布。产品需要保留人工确认环节：

- 用户可直接在当前平台发布稿上编辑。
- 用户可保存当前编辑版本。
- 复制和发布优先使用已保存版本。
- 用户可查看版本历史。
- 用户可回退到历史版本。
- 用户可恢复 AI 自动生成版本。
- 发布前检查中明确提示 AI 稿需要人工确认。

这一设计能降低 AI 误写、夸大表达、平台违规和内容同质化风险。

### 4.7 导出发布包

在真实平台接口暂未接入时，用户仍然需要把生成结果带走。因此系统应支持导出多平台发布包：

- 导出 Markdown 文件。
- 包含四个平台最终发布稿。
- 标记稿件来源是 AI 生成还是人工微调。
- 包含平台策略重点和风险提醒。
- 便于用户复制到各平台后台手动发布。

这个能力可以让 Demo 版本具备真实可用价值，而不完全依赖平台开放接口。

### 4.8 参考样本驱动分析

相比直接爬取平台内容，更合规的 MVP 方式是让用户主动提供参考样本。系统基于这些样本分析可复用规律：

- 标题句式。
- 开头方式。
- 段落节奏。
- 正文结构。
- 标签组合。
- 互动引导。
- 不适合复用的风险表达。

系统必须明确限制：参考样本只用于结构分析，不复制原文句子，不生成与样本高度相似的内容。

未来若接入实时数据，应优先选择官方 API、用户授权数据或用户提供链接，而不是无授权爬取平台内容。

## 5. AI 能力设计

### 5.1 模型接入方式

系统设计为可插拔模型服务，不绑定单一供应商。

可支持：

- 小米相关 AI API。
- OpenAI 兼容 API。
- 国内其他大模型 API。
- 本地规则兜底生成。

比赛版本采用本地 `server.js` 作为 API 代理，前端只请求本地接口，不直接暴露 API Key。代理默认适配 OpenAI-compatible `/chat/completions` 接口；如果小米 API 的协议不同，可替换代理层请求逻辑，前端业务流程不需要变化。

当前代理会把用户主稿、目标平台和参考样本一并发送给模型。模型需要先分析样本结构，再生成原创平台稿。

推荐抽象为：

```text
ModelProvider.generate(prompt, options)
```

不同模型只需要实现统一接口，即可被平台适配器调用。

为了支持爆款分析，可以再拆成两个 AI 能力：

```text
ModelProvider.analyzePlatformSamples(samples, platform)
ModelProvider.generatePlatformDraft(sourceContent, platformStrategy)
```

其中：

- `analyzePlatformSamples` 负责分析平台优质内容，输出标题规律、结构规律、标签规律和风险提醒。
- `generatePlatformDraft` 负责把用户主稿改写成对应平台版本。

这样可以把“分析爆款规律”和“生成用户内容”分开，降低直接抄袭参考内容的风险。

### 5.2 Prompt 输入

模型生成时应包含以下上下文：

- 用户原始标题。
- 用户原始正文。
- 目标受众。
- 目标平台。
- 平台内容风格说明。
- AI 分析出的平台爆款结构。
- AI 分析出的标题模式。
- AI 分析出的标签策略。
- AI 分析出的风险提醒。
- 标题长度限制。
- 正文字数建议。
- 是否生成标签。
- 是否添加互动引导。
- 优质内容结构特征。

### 5.3 输出格式

模型输出应尽量结构化：

```json
{
  "title": "平台标题",
  "summary": "内容摘要",
  "body": ["正文段落1", "正文段落2"],
  "tags": ["标签1", "标签2"],
  "checks": ["需要注意的问题"]
}
```

结构化输出便于前端渲染，也方便后续做检查和编辑。

### 5.4 风险控制

AI 改写需要控制以下风险：

- 不编造事实。
- 不改变用户原意。
- 不直接复制参考内容。
- 不生成明显违规或低质标题。
- 不使用夸大、虚假、诱导性表达。
- 保留用户最终确认权。

## 6. 竞品和差异化

常见内容工具通常只支持单次改写、标题生成或简单排版。CreatorFlow 的差异化在于：

- 以“多平台发布流程”为核心，而不是单点文案生成。
- 每个平台有独立适配器，方便扩展。
- 同时提供生成、预览、检查、复制、发布记录。
- 更适合创作者真实工作流。

## 7. 功能优先级

### P0：必须完成

- 主稿输入。
- 四个平台适配。
- 平台预览。
- 生成平台稿。
- 发布前检查。
- 模拟发布。
- README 和 Demo 说明。

### P1：建议完成

- 字数统计。
- 平台策略说明。
- 复制当前稿。
- 发布记录。
- 产品需求分析文档。

### P2：未来扩展

- 接入大模型 API。
- 接入平台真实发布接口。
- 支持爆款结构模板库。
- 支持历史数据复盘。
- 支持团队协作和审核流。

## 8. 用户流程

```text
输入主稿
  ↓
选择目标受众和语气
  ↓
AI 分析目标平台爆款内容特征
  ↓
生成平台内容策略
  ↓
生成多平台版本
  ↓
查看平台预览和检查项
  ↓
手动微调
  ↓
复制或模拟发布
  ↓
查看发布记录
```

更完整的产品链路是：

```text
用户输入主稿
  ↓
选择平台、受众、语气
  ↓
系统获取平台优质内容样本
  ↓
AI 分析爆款标题、开头、结构、标签和互动方式
  ↓
形成平台策略卡片
  ↓
AI 基于主稿和策略生成各平台版本
  ↓
规则层检查标题长度、字数、标签、风险表达
  ↓
用户预览和手动微调
  ↓
复制、导出或模拟发布
  ↓
记录发布结果
```

## 9. 成功指标

如果产品真实上线，可用以下指标衡量效果：

- 单篇内容多平台适配耗时降低比例。
- 用户复制或发布生成稿的比例。
- 用户手动修改次数。
- 每个平台生成稿通过检查的比例。
- 用户重复使用率。
- Demo 中评委是否能在 1 分钟内理解产品价值。

## 10. 本次比赛版本边界

三天开发周期内，优先保证作品完整可演示：

- 先使用本地规则实现稳定生成。
- 大模型 API 作为架构设计和未来扩展说明。
- 发布功能采用模拟发布。
- 重点展示产品流程、平台差异和扩展能力。

这样的边界更适合比赛交付，也能避免接口申请、平台授权和网络稳定性影响 Demo。
